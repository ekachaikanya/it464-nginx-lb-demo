02 · Forward Proxy

Forward Proxy คืออะไร?

Forward Proxy ทำหน้าที่เป็นตัวแทนของ Client ในการเชื่อมต่อกับ Internet Server ปลายทาง ไม่รู้ว่า Client ตัวจริงคือใคร

DIAGRAM — Forward Proxy Flow
01

🔒 Anonymity & Privacy

ซ่อน IP จริงของ Client จาก Server ปลายทาง ใช้กันมากในองค์กรหรือผู้ที่ต้องการความเป็นส่วนตัว

02

🌍 Bypass Geo-Restriction

เข้าถึงเนื้อหาที่ถูก block ในบางประเทศ โดยส่ง request ผ่าน Proxy ที่อยู่ในประเทศอื่น

03

🏢 Corporate Content Filtering

ควบคุมการเข้าถึงเว็บไซต์ภายในองค์กร block เว็บที่ไม่เหมาะสม และ log การใช้งาน

04

💾 Caching

เก็บ cache ของ response ที่ใช้บ่อย ลด bandwidth และเพิ่มความเร็วในการโหลดสำหรับ request ซ้ำ

03 · Reverse Proxy

Reverse Proxy คืออะไร?

ตรงข้ามกับ Forward Proxy — นี่คือตัวกลางที่นั่งอยู่หน้า Server รับ request จาก Client แล้วส่งต่อไปยัง Backend ที่เหมาะสม

DIAGRAM — Reverse Proxy Flow
💡 Key Point: Client รู้จักแค่ domain เดียว (เช่น https://api.company.com) แต่ด้านหลัง Nginx อาจส่ง request ไปยัง microservice หลายตัว โดย client ไม่รู้เลยว่า backend มีกี่ตัวหรืออยู่ที่ไหน
01

🔒 SSL Termination

จัดการ HTTPS/TLS ที่ Nginx โดย Backend ใช้ HTTP ธรรมดา ลด overhead ของ encryption บน application servers

02

🛡️ Security & Firewall

ซ่อน Backend servers จาก internet โดยตรง rate limiting, IP blocking, และป้องกัน DDoS ที่ layer นี้

03

📦 Caching & Compression

Cache static assets และ compress response ก่อนส่งถึง client ลด bandwidth และเพิ่มความเร็ว

04

🔀 Path-based Routing

ส่ง /api/* ไปยัง API server, /static/* ไปยัง CDN, / ไปยัง Web app — ทั้งหมดบน domain เดียว

04 · Load Balancer

Load Balancer คืออะไร?

Load Balancer คือระบบที่กระจาย traffic ไปยัง Server หลายตัว เพื่อป้องกัน Single Point of Failure และเพิ่ม throughput

DIAGRAM — Load Balancing (Round Robin)

⚙️ Load Balancing Algorithms

🔄

round_robin

ส่ง request ทีละตัว วนเวียน ไปเรื่อยๆ ทุก server ได้รับ request เท่ากัน (default)

⚡

least_conn

ส่งไปยัง server ที่มี active connections น้อยที่สุด เหมาะกับ request ที่ใช้เวลาต่างกัน

🎯

ip_hash

Client IP เดิม → Server เดิมเสมอ ใช้สำหรับ session stickiness

🏋️

weight

กำหนด weight ให้แต่ละ server server แรง → รับ traffic มากกว่า server อ่อน

05 · Nginx Configuration

Nginx Config — ตัวอย่างจริง

Nginx ทำหน้าที่ได้ทั้ง Reverse Proxy และ Load Balancer ในไฟล์ config เดียว

nginx/reverse-proxy.conf
nginx
# ======================================
# Reverse Proxy — ส่ง / ไปยัง app:3000
# ======================================

server {
    listen       80;
    server_name  example.com;

    # Proxy ทุก request ไปยัง backend
    location / {
        proxy_pass         http://app:3000;
        proxy_http_version 1.1;

        # ส่ง header สำคัญไปด้วย
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;

        # Timeout settings
        proxy_connect_timeout 60s;
        proxy_read_timeout    60s;
    }

    # Static files — serve โดยตรงจาก Nginx (เร็วกว่า)
    location /static/ {
        root  /var/www;
        expires 30d;
        add_header Cache-Control "public, no-transform";
    }
}
nginx/load-balancer.conf
nginx
# ======================================
# Load Balancer — กระจาย traffic 3 servers
# ======================================

upstream app_cluster {
    # Algorithm: round_robin (default), least_conn, ip_hash
    least_conn;

    # Backend servers พร้อม weight และ health check
    server app1:8000 weight=3;   # server แรงสุด รับ 3x traffic
    server app2:8000 weight=2;   # medium
    server app3:8000 weight=1;   # server อ่อนสุด

    # ถ้า server ล้ม — ลองใหม่ได้กี่ครั้ง
    # max_fails=3 fail_timeout=30s
    keepalive 32;   # connection pool
}

upstream app_sticky {
    # ip_hash: Client IP เดิม → Server เดิมเสมอ (session sticky)
    ip_hash;
    server app1:8000;
    server app2:8000;
    server app3:8000;
}

server {
    listen 80;

    location / {
        proxy_pass http://app_cluster;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # Health check (Nginx Plus เท่านั้น)
        # health_check interval=10 fails=3 passes=2;
    }

    # Status endpoint สำหรับ monitoring
    location /nginx-status {
        stub_status on;
        allow 127.0.0.1;
        deny all;
    }
}

docker-compose.yml
yaml
version: '3.8'

services:

  # ── NGINX Load Balancer ─────────────────────────────
  nginx:
    image: nginx:alpine
    container_name: nginx-lb
    ports:
      - "80:80"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - app1
      - app2
      - app3
    networks:
      - app-network
    restart: unless-stopped

  # ── App Server 1 ────────────────────────────────────
  app1:
    build: ./app
    container_name: app-server-1
    environment:
      - SERVER_NAME=App Server 1 🟢
      - SERVER_COLOR=#00ff88
    networks:
      - app-network

  # ── App Server 2 ────────────────────────────────────
  app2:
    build: ./app
    container_name: app-server-2
    environment:
      - SERVER_NAME=App Server 2 🔵
      - SERVER_COLOR=#00d4ff
    networks:
      - app-network

  # ── App Server 3 ────────────────────────────────────
  app3:
    build: ./app
    container_name: app-server-3
    environment:
      - SERVER_NAME=App Server 3 🟡
      - SERVER_COLOR=#ffd700
    networks:
      - app-network

networks:
  app-network:
    driver: bridge
nginx/nginx.conf
nginx
# ================================================
# Nginx — Load Balancer + Reverse Proxy
# ================================================

upstream backend {
    least_conn;          # Algorithm: ส่งไป server ที่ว่างสุด

    server app1:5000;
    server app2:5000;
    server app3:5000;
}

server {
    listen 80;
    server_name localhost;

    # ── Main route → Load Balanced ──────────────
    location / {
        proxy_pass         http://backend;
        proxy_http_version 1.1;
        proxy_set_header   Connection "";
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # ── Nginx Status Monitoring ──────────────────
    location /status {
        stub_status on;
        access_log off;
    }
}
app/app.py
python
import os, socket
from flask import Flask, jsonify

app = Flask(__name__)

SERVER_NAME  = os.getenv('SERVER_NAME', 'Unknown Server')
SERVER_COLOR = os.getenv('SERVER_COLOR', '#ffffff')
HOSTNAME     = socket.gethostname()

# ────────────────────────────────────────────
# GET / — แสดงว่า request นี้ไปถึง server ไหน
# ────────────────────────────────────────────
@app.route('/')
def index():
    html = f"""
    <!DOCTYPE html><html>
    <head>
      <title>{SERVER_NAME}</title>
      <style>
        body {{ font-family: monospace; background: #060b14;
               color: #c8d8f0; display: flex; align-items: center;
               justify-content: center; height: 100vh; margin: 0; }}
        .card {{ text-align: center; padding: 40px;
                 border: 2px solid {SERVER_COLOR};
                 border-radius: 16px; background: #0d1626; }}
        h1 {{ color: {SERVER_COLOR}; font-size: 2rem; margin-bottom: 12px; }}
        .hostname {{ color: #546e7a; font-size: 0.9rem; }}
        .refresh {{ margin-top: 20px; color: #546e7a; font-size: 0.8rem; }}
      </style>
    </head>
    <body>
      <div class="card">
        <h1>⚡ {SERVER_NAME}</h1>
        <p class="hostname">Hostname: {HOSTNAME}</p>
        <p class="refresh">🔄 กด F5 เพื่อดู Load Balancing ในแอคชัน!</p>
      </div>
    </body></html>
    """
    return html

# Health check endpoint
@app.route('/health')
def health():
    return jsonify({
        'status': 'ok',
        'server': SERVER_NAME,
        'hostname': HOSTNAME
    })

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
